# 文档维护方法论

本文件定义了 AI 协作文档体系的维护方式，目标是让文档自我演进、持续精简。

---

## 1. 维护目标

- AI 犯错越来越少。
- 团队重复解释越来越少。
- `AGENTS.md` 始终简短（200 行以内）。
- 专题文档始终可读、可执行。
- 不是让文档越来越多，而是让文档越来越准。

---

## 2. 规则的 7 种来源

| 来源 | 举例 |
|------|------|
| 线上 Bug | AI 生成的代码导致运行时错误 |
| Code Review | 发现 AI 反复犯同一类错误 |
| 重复沟通 | 用户多次纠正同一问题 |
| 项目重构 | 目录结构变更、架构升级 |
| 公共组件沉淀 | 新增全局组件需要注册规范 |
| 用户明确要求 | 用户说"以后都要这样做" |
| 团队约定 | 代码风格统一、命名规范 |

---

## 3. 规则推荐判断

讨论新规则时先判断以下信息，实际写入文档时只保留当前有效结论：

```markdown
### 规则标题

- **来源**：[哪次任务 / 什么问题触发]
- **适用范围**：[哪些场景需要遵守]
- **规则内容**：[具体要求]
- **原因**：[为什么需要这条规则]
- **例外**：[什么情况下可以不遵守]
- **示例**：[正确 vs 错误的代码示例]
```

来源和讨论过程不默认写入长期规则正文；只有能解释重要架构取舍时才写入 `docs/ai/decisions.md`。

---

## 4. 规则沉淀路径

```
聊天或任务中的临时约定
  ↓ 经过 2-3 次真实任务验证
docs/ai/ 中的候选规则
  ↓ 仍然稳定、重复出现、值得长期复用
docs/ 专题文档
  ↓ 已成为全局红线或高频入口
AGENTS.md
```

- 候选规则先放在 `docs/ai/`，用真实任务验证价值。
- 规则稳定后再进入 `docs/` 或 `AGENTS.md`。
- 规则过时后直接更新或删除，避免依赖显式状态标签制造误导。

---

## 5. 瘦身原则

| 原则 | 说明 |
|------|------|
| 能靠代码解决的不写文档 | 比如 ESLint 规则、TypeScript 类型约束 |
| 能靠类型约束的不写文档 | 比如 Props 接口定义 |
| 能放专题文档的不放 AGENTS.md | AGENTS.md 只留全局红线 |
| 能用示例说明的不写长篇 | 一个代码示例胜过一段文字 |
| 一次性约定不进入规则 | 只有反复需要的才值得沉淀 |

---

## 6. 定期瘦身建议

建议每 2-4 周审查一次文档体系，检查：

- [ ] 是否有规则在多个文档中重复出现？→ 合并到一处
- [ ] 是否有规则已过时（项目代码已变更）？→ 直接更新或删除，不保留误导性状态痕迹
- [ ] 是否有规则过于细节化？→ 移入代码注释或删除
- [ ] `AGENTS.md` 是否超过 200 行？→ 下沉细节到专题文档
- [ ] 是否有规则可以通过自动化工具强制执行？→ 迁移为代码约束

---

## 7. 推荐闭环流程

```
做任务
  ↓
输出实现 + 复盘
  ↓
判断是否有可沉淀的规则
  ↓
先进入 docs/ai/ 作为候选规则
  ↓
经过 2-3 次任务验证
  ↓
确认仍然稳定，再进入 docs/ 或 AGENTS.md
  ↓
定期瘦身审查
```

**关键原则**：
- AI 只提建议，不自动修改文档（除非用户明确要求）。
- 新规则先进入 `docs/ai/`，验证后再决定去向。
- 规则来自真实问题，不提前假设。
