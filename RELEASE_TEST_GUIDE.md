# Monster Tools 发布与整体更新测试指南

本指南用于验证和测试 `Monster Tools` 客户端的本地开发热重载（HMR）、自动版本同步、一键发布流水线、CI/CD 构建以及 Tauri 原生整体自动更新功能，确保每一次发布都稳定可靠。

---

## 一、 目录结构规范与检查

发布或测试前，请先确保项目目录的整洁与规范：

*   **根目录**：仅应保留前端配置（Vite, Tailwind, TS 配置等）、[README.md](file:///README.md) 使用文档、开源协议 [LICENSE](file:///LICENSE) 以及此测试指南。
*   **规则约束**：确保 [AGENTS.md](file:///AGENTS.md) 处于最新状态，以便约束后续任何 AI 协作编码工具的开发行为。
*   **开发目录 `src`**：只存放 Vue 3 + TypeScript 的前端业务代码。
*   **后端目录 `src-tauri`**：只存放 Tauri 配置和精简后的 Rust 代码（确保 `Cargo.toml` 中无冗余废弃的依赖库）。
*   **脚本目录 `scripts`**：
    *   `sync-version.js`：版本自动同步服务。
    *   `changelog.js`：交互式双语更新日志自动生成器。
    *   `release.js`：一键自动化发布流工具。

---

## 二、 本地开发与启动（HMR）验证

### 1. 启动本地开发服务
在终端执行以下命令：
```bash
npm run tauri:dev
```

### 2. 验证 HMR（前端热重载）
*   打开 `src/App.vue`，尝试随意修改一个文本（例如，将主页的 `Tauri v2 + Vue 3 整体更新客户端` 稍作修改并保存）。
*   **预期结果**：Tauri 调试窗口内的页面应该在保存的瞬间**自动刷新并应用新文本**，且无需重新编译 Rust。

### 3. 验证窗口尺寸缩放限制
*   手动拖拽拉伸 Tauri 窗口，将其缩到最小。
*   **预期结果**：窗口缩小到一定程度后无法再缩小。使用截图或量尺工具确认窗口最小极限为 **`900 × 630`**，满足最小设计尺寸限制。

## 三、 一键自动发布流水线验证

发布新版本前，需要全自动运行各项提问与同步机制，可以使用以下命令进行完整验证：

### 1. 触发一键发布
在控制台输入：
```bash
npm run release
```

### 2. 检查 SemVer 版本选择
*   终端应当显示当前的 package 版本，并提供 4 个选项：
    *   `1) Patch`
    *   `2) Minor`
    *   `3) Major`
    *   `4) 自定义版本号`
*   输入 `1`（或者指定一个版本）并按回车确认。

### 3. 验证版本同步
*   **检查文件 1**：打开 `package.json`，验证其 `"version"` 字段已自动修改为新的版本。
*   **检查文件 2**：打开 `src-tauri/tauri.conf.json`，验证其 `"version"` 字段已与 package 保持完全一致。
*   **检查文件 3**：打开 `src-tauri/Cargo.toml`，验证其 `version = "..."` 字段已被同步更新。

### 4. 验证交互式双语日志与暂存
*   脚本应该打印最近 5 次 Git 提交作为编写参考。
*   依次在提示下输入本次发布的**中文日志**与**英文日志**（多行以回车换行，空行直接按回车确认结束）。
*   **预期结果**：
    1.  [CHANGELOG.zh-CN.md](file:///CHANGELOG.zh-CN.md) 顶部已被追加了格式化的中文发布日志。
    2.  [CHANGELOG.md](file:///CHANGELOG.md) 顶部已被追加了格式化的英文发布日志。
    3.  `dist/release-notes.md` 已单独输出，且仅包含本次英文变更描述。
*   **检查 Git 暂存**：在终端运行 `git status`，验证本次版本修改和日志生成的文件都已被自动添加进 Git 暂存区。
*   **检查 Git Tag**：运行 `git log -n 1` 以及 `git tag`，确认本地已成功创建了一个对应于此版本的 `v<版本号>` 的 Git 标签。

## 四、 CI/CD 自动构建与 Release 验证

每次推送带版本 Tag 的提交，都会在 GitHub 远端触发编译，请依以下步骤进行检验：

### 1. 推送到远端仓库
当发布脚本询问是否立即推送时，选择 `y`，或者手动在终端执行：
```bash
git push && git push origin --tags
```

### 2. 检查 GitHub Actions 运行状态
*   打开您的 GitHub 仓库页面，点击 **Actions** 板块。
*   **预期结果**：应该自动触发了一个名为 `release` 的工作流，正在并行编译 Windows、macOS 和 Ubuntu 三个系统的原生安装包。

### 3. 验证 GitHub Release
*   Actions 运行完毕后，检查仓库的 **Releases** 页面。
*   **预期结果**：
    1.  自动生成了一个名为 `Monster Tools v<版本号>` 的正式发布（或草稿）。
    2.  Release 的说明文字（Body）与您在发布脚本中输入的英文更新日志完全一致。
    3.  构建产物中包含各系统的安装包（如 Windows 的 `.msi` 或 `.exe` 等）以及用于更新的安全签名校验文件。

## 五、 原生整体更新 (Tauri Updater) 验证

### 1. 安装旧版本
*   在一台未安装该应用（或安装了老版本）的机器上，下载并安装您的老版本（如 `v0.1.0`）客户端。

### 2. 检查更新检测
*   启动客户端，在主页上点击 **“检查应用更新”** 按钮。
*   **预期结果**：
    *   页面显示 “正在检查应用更新...”。
    *   如果发现远端发布了更新的版本，应提示发现新版本并开始下载，下载过程中能实时观察到进度条与百分比数值增加。
    *   下载安装完成后，客户端应**自动重启**，并以最新的版本启动，主页版本显示正常更新后的版本。
